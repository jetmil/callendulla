# AGPL-3.0 §13 — что обязан делать оператор сервиса

Callendulla распространяется под AGPL-3.0. Если ты её **просто
запускаешь у себя дома для своей семьи** — никаких обязательств перед
посторонними нет.

Но как только ты даёшь к ней доступ **третьим лицам через сеть** —
неважно, бесплатно или за деньги — вступает в силу **параграф 13
AGPL-3.0**:

> Notwithstanding any other provision of this License, if you modify
> the Program, your modified version must prominently offer all users
> interacting with it remotely through a computer network ... an
> opportunity to receive the Corresponding Source of your version
> ... from a network server at no charge ...

Перевод человеческим языком:

- Если ты **модифицировал** код Callendulla и хостишь модифицированную
  версию как сервис для других людей —
- ты **обязан** дать этим людям возможность получить твой исходный код
  с тем же набором правок.

Это работает на любом дистанционном использовании, не только на классическом
SaaS-клоне: Telegram-бот для друзей, открытая регистрация в твоём
инстансе, веб-консоль на твоём поддомене — всё это network use.

## Что Callendulla делает «из коробки» для compliance

Сервер шлёт два сигнала о источнике кода в каждом HTTP-ответе и в одном
явном endpoint'е:

1. **HTTP header `X-Source-URL`** на каждом ответе API. Значение
   читается из env `AGPL_SOURCE_URL`.
2. **`GET /source`** возвращает JSON с URL и SHA коммита, на котором
   собран запущенный образ:
   ```json
   {
     "license": "AGPL-3.0",
     "source_url": "https://github.com/jetmil/callendulla",
     "commit_sha": "abc1234",
     "build_date": "2026-05-12T05:47:00Z"
   }
   ```
3. В Telegram-боте команда `/source` отвечает теми же данными в чат.

> **Реализация частично WIP** — middleware и endpoint запланированы в
> одном из первых runtime-коммитов. Если ты деплоишь до того момента —
> поставь заглушки сам, иначе формально нарушаешь §13.

## Если ты форкнул и модифицировал

**Минимум, чтобы оставаться compliant:**

### 1. Замени `AGPL_SOURCE_URL` на твой fork

```bash
# .env
AGPL_SOURCE_URL=https://github.com/yourname/callendulla-fork
```

Без этого юзеры твоего сервиса видят ссылку на оригинальный репо —
а там нет твоих правок. Это **нарушение §13**.

### 2. Держи свой fork публичным

Если репо приватный, юзеры не могут получить source. Сделай его public,
или хотя бы открой публичный mirror.

### 3. Распространяй модификации под той же AGPL-3.0

§13 не разрешает «дополнить и сделать closed-source». Все твои правки
должны быть под AGPL-3.0 (или совместимой более поздней версией).

### 4. Сохрани LICENSE и copyright headers в коде

Не удаляй файл `LICENSE` из репо или Docker image. Не вычищай AGPL
headers из исходников.

### 5. Сохрани `/source` endpoint и `X-Source-URL` header

Не убирай middleware и endpoint, через которые юзеры узнают о твоём
коде. Можно поменять путь endpoint'а (например на `/agpl-source`), но
эквивалентный механизм должен работать.

## Что **можно** делать без обязательств перед посторонними

- Использовать у себя дома для своей семьи (это не network service для
  «третьих лиц»)
- Запускать на работе для коллег **внутри** организации (одна
  организация = не публичный сервис)
- Модифицировать код «в стол» — обязательства возникают только при
  network-распространении

## Если AGPL мешает твоему сценарию

Если ты компания, которая хочет:

- Встроить Callendulla в closed-source продукт
- Не открывать модификации
- Защититься от обязательств §13 для конечных клиентов

— это **возможно** через коммерческую лицензию. Контакт:
**jetmil@proton.me**.

Это стандартная dual-licensing практика (как у MongoDB до BSL, GitLab
CE/EE, Nextcloud Enterprise). Без CLA от контрибьюторов dual-licensing
не работает — поэтому в `CONTRIBUTING.md` указан CLA-требование.
